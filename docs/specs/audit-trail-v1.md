# Title

Audit Trail v1

# Status

Draft

# Context

The PRD requires:

- auditability by design
- auditable events for all sensitive operations
- an audit log as a core deliverable
- plugin releases and operational changes to be traceable and auditable

ADR-001 places audit trail in the core as shared platform infrastructure. ADR-004 requires auditability for sensitive authorization changes. ADR-006 and ADR-008 establish that plugin lifecycle and public extension events are core-governed platform contracts. The platform therefore needs a shared audit model that:

- records sensitive operations performed by the core and plugins
- remains separate from business-domain history or workflow notes
- preserves tenant context and actor context
- supports investigation, compliance traceability, and operational review

# Specification

## 1. Objectives

This specification defines:

- what the audit trail is for
- which operations must create audit records
- the minimum shape of an audit record
- how plugins participate in audit logging
- safety and retention rules for audit data

## 2. Scope of the Audit Trail

The audit trail is the append-only record of sensitive platform and plugin operations.

The audit trail is not the same as:

- debug logs
- application metrics
- domain workflow comments
- business object change history stored for end-user collaboration

Those may coexist, but the audit trail is the authoritative security and governance record.

## 3. What Must Be Audited

At minimum, v1 must audit:

- authentication success and failure where platform policy deems it sensitive
- principal access changes
- membership changes
- role and permission grant changes
- organization and scope lifecycle changes
- plugin install, enable, disable, upgrade, and uninstall attempts and results
- configuration changes with security, authorization, tenancy, storage, or integration impact
- attachment upload, download authorization-sensitive access, replacement, and deletion where applicable
- API or UI operations classified as sensitive by the core or by a plugin contract
- administrative operations on identity-to-actor linkages

Plugins must be able to emit audit records for their own sensitive operations through core contracts.

## 4. Audit Record Model

Each audit record should contain, at minimum:

- stable audit identifier
- timestamp
- event type
- event outcome
- origin component
- actor context
- tenancy context
- target context
- summary payload
- correlation metadata

### Event Type

The event type identifies what happened in stable semantic form.

Recommended naming pattern:

- `core.plugins.enabled`
- `core.permissions.grant.created`
- `core.organizations.updated`
- `plugin.assets.asset.deleted`

Rules:

- event types must be stable and namespaced by owning component
- event type naming must describe meaning, not implementation details

### Event Outcome

The event outcome expresses whether the operation succeeded or failed.

Minimum values in v1:

- `attempt`
- `success`
- `failure`

If the platform records only final state for a given operation, `success` and `failure` are still required.

### Origin Component

The origin component identifies whether the record was generated by:

- the core
- a specific plugin
- a platform subsystem acting on behalf of another component

### Actor Context

Actor context should capture:

- principal identifier if known
- membership identifier if relevant
- authentication provider or execution origin if relevant
- system or background-job indicator when no interactive principal exists

Actor context is about who triggered the operation, not who owns the business object.

### Tenancy Context

Tenancy context should capture:

- organization identifier when applicable
- scope identifier when applicable

Platform-global operations may omit organization and scope identifiers, but tenant-bound sensitive operations must carry them.

### Target Context

Target context identifies what the operation acted upon.

Examples:

- plugin identifier
- permission key
- organization identifier
- scope identifier
- attachment identifier
- domain object reference

### Summary Payload

The summary payload contains the minimum structured facts needed to understand the change.

Examples:

- changed fields
- previous and new high-level state
- affected permission or role key
- plugin version transition

Payload rules:

- payload must stay structured
- payload must avoid raw secrets, credentials, tokens, and sensitive file contents
- payload must prefer identifiers and redacted values over full confidential data

### Correlation Metadata

Correlation metadata should support traceability across request or job boundaries.

Recommended fields:

- request identifier
- job identifier
- event or workflow correlation identifier
- source IP or client metadata where policy allows

## 5. Append-Only and Integrity Rules

The audit trail must be append-only in normal platform operation.

Rules:

- audit records must not be updated or deleted silently
- correction or enrichment should happen through explicit compensating audit records or formally governed maintenance operations
- audit storage must support ordering by timestamp and stable identifier

## 6. Redaction and Sensitive Data Rules

The audit trail exists to improve accountability, not to leak secrets.

The platform must not store in audit payloads:

- plaintext secrets
- passwords
- raw tokens
- private keys
- full sensitive attachment content
- unnecessary personal data beyond what the event needs for traceability

Recommended strategy:

- store identifiers
- store redacted field names and high-level deltas
- store hashes or fingerprints where proving change matters more than storing content

## 7. Plugin Participation Rules

Plugins must use the core audit contract for sensitive operations.

Rules:

- plugins may define their own namespaced audit event types
- plugins must not create isolated private audit systems for platform-significant actions
- plugin audit records must carry plugin origin and, when relevant, organization and scope context
- disabling a plugin must not destroy the integrity of previously stored audit records

## 8. Audit Access and Export

The core must expose a governed way to inspect and export audit records.

Minimum v1 rules:

- viewing audit records requires a dedicated permission such as `core.audit-logs.view`
- exporting audit records requires a stronger permission such as `core.audit-logs.export`
- exports must use structured formats such as NDJSON or CSV
- filters should at minimum support event type, outcome, origin component, actor context, tenancy context, target context, execution origin, and time window
- export endpoints and commands must reuse the same append-only audit source as the normal audit view

## 9. Relationship to Domain History

A plugin may also maintain user-facing domain history, but that is separate from the audit trail.

Examples:

- a risk plugin may keep review history for a risk record
- a policy plugin may keep approval comments
- an evidence plugin may keep validation notes

Those histories do not replace audit records for the underlying sensitive operations.

## 9. Access to Audit Records

Audit log access is itself sensitive.

Rules:

- audit-log viewing must be protected by explicit permissions
- export or bulk access to audit records should use stricter permissions or policy checks if required
- plugins must not assume unrestricted read access to core or peer-plugin audit data

Recommended baseline permissions:

- `core.audit_logs.view`
- `core.audit_logs.export`

## 10. Retention and Lifecycle

The platform must define retention behavior explicitly.

Baseline v1 policy:

- audit records are retained by default
- archival is preferred over destructive deletion
- purge operations, if later allowed, must themselves be auditable and policy-governed

## 11. Operational Examples

Examples of auditable events:

- enabling plugin `identity-local`
- revoking a membership from organization `ORG-A`
- changing an organization default locale
- exporting a report containing tenant-sensitive data
- deleting an attachment from a risk record
- linking a principal to a functional actor

## 12. Integration with Events and Observability

The audit trail is related to, but distinct from, the event bus and observability stack.

Rules:

- not every public event must create an audit record
- every sensitive auditable operation should have an audit outcome regardless of whether public events are emitted
- metrics and debug logs do not satisfy audit requirements by themselves

# Consequences

- The core and plugins get one consistent traceability model for sensitive operations.
- Tenant context, actor context, and target context can be investigated together.
- Future domain plugins can build user-facing history without confusing it with security-grade audit records.
- The platform will need explicit audit-writing contracts, redaction policy, and query permissions.
- Storage and retention policy become architectural concerns early, which is appropriate given the PRD’s traceability requirements.
