# Title

Artifacts and Attachments v1

# Status

Draft

# Context

The PRD requires the core to provide base storage for attachments and artifacts so that plugins can attach evidence, exports, reports, and governance records without inventing their own storage layer. ADR-001 places this capability in the core. ADR-010 requires sensitive handling of evidence-related actions to be auditable. ADR-012 and the domain plugins depend on this capability for workflows, reviews, and evidence traceability.

The platform therefore needs a minimal artifact subsystem that:

- stores private files and metadata through one shared core contract
- remains domain-agnostic
- preserves tenancy context
- integrates with audit and public events
- can be reused by multiple plugins such as controls, risks, findings, and privacy

# Specification

## 1. Objectives

This specification defines:

- what the v1 artifact subsystem provides
- how files and metadata are modeled
- how plugins should store domain evidence
- what remains out of scope for later versions

## 2. Core Scope

In v1, the core artifact subsystem should provide:

- private file storage on a configured disk
- artifact metadata persistence in the shared database
- subject binding to a domain object or process
- tenancy-aware filtering
- integrity metadata such as file size and content hash
- audit trail records for artifact creation
- public event publication for artifact creation

The core does not yet provide:

- antivirus or content scanning
- OCR, parsing, or document indexing
- signed public download URLs
- version chains or check-in/check-out semantics
- retention policies or legal-hold mechanics

## 3. Generic Artifact Model

An artifact is a stored file plus metadata describing what it belongs to.

In v1, an artifact must record:

- stable artifact identifier
- owner component
- subject type
- subject identifier
- artifact type
- label
- original filename
- media type
- file extension
- size in bytes
- content hash
- storage disk
- storage path
- organization identifier
- optional scope identifier
- optional principal and membership references
- optional metadata payload
- creation timestamp

## 4. Subject Binding Rules

Artifacts are always attached to a generic subject reference.

Rules:

- the core stores only generic `subject_type` and `subject_id`
- the meaning of the subject belongs to the owning plugin
- the core does not define domain-specific evidence semantics
- plugins must use stable subject identifiers so audit, workflow, and evidence remain traceable

## 5. Storage and Security Rules

Artifacts must be stored defensively.

Rules:

- v1 storage is private by default
- file paths must be generated by the core, not by plugins
- original filenames are metadata only and must not be trusted as storage paths
- the core should capture a content hash for later integrity checks
- plugins must not bypass the core artifact service when storing shared platform evidence
- uploads must be validated against an explicit allowed file profile before artifact storage
- the platform must not accept arbitrary executable or active-content file types by default
- upload type validation must use both allowed extensions and server-side detected content type
- artifact download and preview behavior must remain conservative for unknown or non-previewable file types

See [`file-upload-security-and-type-validation-v1.md`](/media/marc/4T_EXFAT/web/pymesec.com/pymesec/docs/specs/file-upload-security-and-type-validation-v1.md) for the platform-wide upload baseline.

## 6. Tenancy Rules

Artifacts are tenancy-aware.

Rules:

- tenant-bound artifacts must carry an organization identifier
- artifacts may also carry a scope identifier
- queries must allow filtering by organization and scope
- plugins must not attach tenant evidence without tenant context unless the subject is explicitly platform-scoped

## 7. Audit and Event Rules

Artifact creation is a traceable operation.

Rules:

- artifact creation should emit `core.artifacts.created`
- artifact creation should generate an audit record when performed through platform or plugin flows
- audit records should preserve principal, membership, organization, and scope where available

## 8. Plugin Responsibilities

Plugins may:

- upload artifacts for their domain subjects
- classify artifacts through artifact type and metadata
- render evidence lists and review boards using artifact metadata

Plugins must not:

- create their own parallel private storage model for shared evidence
- assume public file access
- encode business semantics into the core metadata schema

## 9. Minimal v1 Use Cases

The subsystem should be sufficient for:

- attaching evidence to controls
- storing review exports or generated reports
- associating findings or risks with supporting files
- listing artifacts per subject in shell screens

## 10. Out of Scope in v1

This v1 specification does not yet define:

- artifact download and streaming UX
- artifact deletion and retention workflows
- evidence versioning
- cross-artifact relationships
- encrypted per-tenant keys

# Consequences

- The platform gets one shared evidence substrate for domain plugins.
- Controls, risks, findings, privacy, and continuity plugins can all attach files without inventing incompatible storage.
- Auditability and event publication remain aligned with the rest of the core runtime.
- Later work can extend this subsystem toward richer evidence lifecycle management without changing the basic plugin contract.
